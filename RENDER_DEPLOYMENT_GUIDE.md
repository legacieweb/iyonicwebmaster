# Render Deployment Guide for Iyonicorp Web Application

This guide will walk you through deploying your Iyonicorp web application on Render.

## Prerequisites

1. A Render account (sign up at https://render.com)
2. Your code pushed to a Git repository (GitHub, GitLab, or Bitbucket)
3. A PostgreSQL database (you already have Neon DB configured)

## Step-by-Step Deployment

### Step 1: Push Your Code to Git

Make sure your code is committed and pushed to your Git repository:

```bash
git add .
git commit -m "Add render.yaml for deployment"
git push origin main
```

### Step 2: Connect to Render

1. Go to [Render Dashboard](https://dashboard.render.com)
2. Click **"New +"** button
3. Select **"Blueprint"** (this uses your render.yaml file)

### Step 3: Connect Your Repository

1. Connect your GitHub/GitLab/Bitbucket account if not already connected
2. Select your repository containing the iyonicweb project
3. Click **"Connect"**

### Step 4: Configure Environment Variables

Render will automatically detect your `render.yaml` file. You'll need to configure these environment variables:

#### Required Environment Variables:

| Variable | Description | Where to Get It |
|----------|-------------|-----------------|
| `DATABASE_URL` | PostgreSQL connection string | Your Neon DB dashboard |
| `JWT_SECRET` | Secret key for JWT tokens | Auto-generated by Render |
| `EMAIL_USER` | Gmail address for sending emails | Your Gmail account |
| `EMAIL_PASS` | Gmail app password | Google Account settings |
| `VITE_PAYSTACK_PUBLIC_KEY` | Paystack public key | Paystack dashboard |
| `VITE_ADMIN_EMAIL` | Admin email address | Your admin email |
| `VITE_ADMIN_PASSWORD` | Admin password | Your admin password |

#### How to Set Environment Variables:

1. In Render dashboard, go to your service
2. Click **"Environment"** in the left sidebar
3. Add each variable:
   - Click **"Add Environment Variable"**
   - Enter the key (e.g., `DATABASE_URL`)
   - Enter the value
   - Click **"Save"**

### Step 5: Deploy

1. Click **"Create Blueprint Instance"**
2. Render will:
   - Install dependencies (`npm install`)
   - Build the frontend (`npm run build`)
   - Start the server (`npm run server`)
3. Wait for the deployment to complete (usually 2-5 minutes)

### Step 6: Access Your Application

Once deployed, Render will provide you with a URL like:
```
https://iyonicweb.onrender.com
```

## Important Notes

### Database Setup

Your application uses Neon DB (PostgreSQL). Make sure:

1. Your database is accessible from Render's servers
2. The `DATABASE_URL` is correctly formatted
3. SSL is enabled (already configured in your code)

### File Uploads

Render's free tier has ephemeral storage, meaning uploaded files will be lost on redeployment. For production:

1. Consider using a cloud storage service (AWS S3, Cloudinary, etc.)
2. Or upgrade to a paid Render plan with persistent disk

### Environment Variables Security

- Never commit `.env` files to Git
- Use Render's environment variable management
- Generate a strong `JWT_SECRET` for production

### Custom Domain (Optional)

To use a custom domain:

1. Go to your service in Render dashboard
2. Click **"Settings"**
3. Scroll to **"Custom Domains"**
4. Add your domain and follow DNS configuration instructions

## Troubleshooting

### Build Fails

- Check the build logs in Render dashboard
- Ensure all dependencies are in `package.json`
- Verify Node.js version compatibility

### Dependency Conflicts

If you see errors like `ERESOLVE could not resolve` or peer dependency conflicts:

The `render.yaml` is configured to use `--legacy-peer-deps` flag to handle the `react-paystack` package which doesn't fully support React 19. This is safe and won't affect functionality.

If you encounter other dependency conflicts, you can:
1. Update the conflicting package to a newer version
2. Or add `--legacy-peer-deps` to the build command in `render.yaml`

### Database Connection Issues

- Verify `DATABASE_URL` is correct
- Check if your Neon DB allows connections from Render's IPs
- Ensure SSL is enabled in the connection string

### Application Crashes

- Check the logs in Render dashboard
- Verify all environment variables are set
- Test locally with production environment variables

### Health Check Fails

The application includes a health check endpoint at `/api/health`. If this fails:

1. Check if the server is starting correctly
2. Verify the port configuration
3. Check database connectivity

## Scaling

### Free Tier Limitations

- Service spins down after 15 minutes of inactivity
- Limited to 750 hours/month
- No persistent disk

### Upgrading to Paid Plan

For production use, consider upgrading to:

- **Starter**: $7/month - Always on, custom domains
- **Standard**: $25/month - More resources, better performance
- **Pro**: $85/month - High availability, priority support

## Automatic Deployments

Render automatically deploys when you push to your main branch. To disable:

1. Go to service settings
2. Toggle **"Auto-Deploy"** off

## Rollbacks

If a deployment fails:

1. Go to your service in Render dashboard
2. Click **"Events"** tab
3. Find the last successful deployment
4. Click **"Rollback to this deploy"**

## Quick Reference

### Your render.yaml Configuration

```yaml
services:
  - type: web
    name: iyonicweb
    runtime: node
    plan: free
    buildCommand: npm install && npm run build
    startCommand: npm run server
    envVars:
      - key: NODE_ENV
        value: production
      - key: PORT
        value: 10000
      - key: DATABASE_URL
        sync: false
      - key: JWT_SECRET
        generateValue: true
      - key: EMAIL_USER
        sync: false
      - key: EMAIL_PASS
        sync: false
      - key: VITE_PAYSTACK_PUBLIC_KEY
        sync: false
      - key: VITE_ADMIN_EMAIL
        sync: false
      - key: VITE_ADMIN_PASSWORD
        sync: false
    healthCheckPath: /api/health
    autoDeploy: true
```

### Key Commands

```bash
# Test build locally
npm run build

# Test server locally
npm run server

# Test full application locally
npm run dev
```
